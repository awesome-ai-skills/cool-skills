# HTML Experience Design

Use this contract for every self-contained HTML architecture report generated by this skill. The report should feel like an algorithm-engineering cockpit: visually distinctive, fast to scan, comfortable for long reading, and grounded in source code.

Render repository facts, pipeline stages, flow nodes, statistics, and source anchors from the shared `algorithm-repo-analysis.json` manifest. The Markdown report may inform prose editing, but it is not the HTML data source.

## 1. Experience Objective

Optimize for a new engineer answering these questions quickly:

- What domain and industrial stage does this repository cover?
- What is the primary pipeline, and which stages are absent?
- Which models, layers, trainers, examples, and tests matter most?
- How does one request or training sample move through the code?
- What should I read first for my goal, and what can I safely ignore initially?
- Where should I start when adding a model, feature, loss, dataset, or metric?
- How can I open or copy the exact source location?

Build a usable reading tool, not a marketing landing page and not a Markdown document with decorative cards. Preserve all core report content while progressively disclosing dense details.

## 2. Visual System

- Use CSS custom properties for light and dark themes: page, surface, elevated surface, text, muted text, border, shadow, focus, code, and semantic stage colors.
- Use a neutral base plus distinct semantic accents for data, recall, pre-ranking, ranking, serving, and limitations. Do not let one hue dominate the whole page.
- Create visual interest through hierarchy, topology, state, and restrained motion. Do not use decorative gradient blobs, bokeh, oversized empty heroes, glass effects that reduce contrast, or ornamental 3D.
- Use the system sans-serif stack and a system monospace stack. Keep letter spacing at `0`.
- Use compact type inside controls and panels. Reserve the largest type for the repository name or literal report title.
- Keep component radii at `8px` or less, except circular indicators and compact pills.
- Use self-contained inline SVG icons for familiar actions such as search, menu, theme, copy, source, external editor, expand, and back-to-top. Do not use emoji as control icons.
- Give code snippets a terminal-like dark surface, but render paths and symbols as interactive source chips rather than making all technical content look like code blocks.

## 3. Page Architecture

The page must contain these functional regions:

| Region | Required behavior |
|---|---|
| App toolbar | Repository identity, search trigger/input, and theme control |
| Overview | One-sentence framing, domain, framework, model count, measured language/code-volume statistics, pipeline coverage, report/source timestamp when known |
| Navigation | Active-section state with `aria-current="location"`, smooth scrolling, persistent across desktop widths |
| Pipeline | Interactive stage navigation plus a synchronized detail panel |
| Data flows | Separate visual flows for offline training, online request/inference, and representative model `forward()` when applicable |
| Model atlas | Searchable/filterable inventory grouped by role, family, or pipeline stage |
| Code Map | Paths, symbols, categories, stages, source actions, and line numbers when discoverable |
| Learning route | Goal-oriented ordered reading tracks with question, depth, prerequisites, and source anchors |
| Change guide | Common modification intents mapped to extension points, related code, tests, and contracts |
| Assessment | Strengths, limitations, production gaps, and recommended newcomer reading order |

Add a thin reading-progress indicator when the report is long. Keep a back-to-top action available after the first viewport.

## 4. Pipeline Dashboard

- Render the industrial pipeline horizontally at desktop widths. Use deliberate internal scrolling when necessary; do not collapse the pipeline into an unstructured paragraph.
- Use semantic `button` or tab patterns with `aria-selected`, `aria-controls`, and roving keyboard focus. Support `ArrowLeft`, `ArrowRight`, `Home`, and `End` where tabs are used.
- Every stage node must show stage name, coverage state, and a short role label. Coverage must remain understandable without color alone.
- The synchronized panel must show industrial goal, coverage, core models, precise paths/symbols, implementation logic, inputs/outputs, and what the repository does not cover.
- Update the active state immediately. If a transition timer is used, cancel the previous timer before starting another to prevent stale panel content.
- Preserve a useful no-JavaScript fallback containing the stage mapping table or expanded stage details.

## 5. Data-Flow Diagrams

- Use native HTML/CSS step cards, connected nodes, swimlanes, or timelines. Do not use a terminal-style text block as the primary flow.
- Distinguish offline, online, and model-internal flows with headings and semantic color accents.
- Each node must contain a concise operation, its input/output meaning, and a source anchor such as `path/to/file.py::Class.forward`.
- Make long flows pan or wrap deliberately. Arrows, connectors, labels, and node order must remain coherent at every target viewport.
- Allow dense node details to expand without resizing unrelated controls or causing layout jumps.

## 6. Model and Code Exploration

Before the searchable inventory, include a compact engineering-scale view when repository statistics are available:

- Show primary implementation language, analyzed files, total measured code lines, and repository revision/method.
- Render the language distribution with native HTML/CSS bars or rows. Every bar must have visible file/line values and a text label; meaning cannot depend on width or color alone.
- Render role-based source scale for models, layers/operators, data/input, training/inference, examples, and tests when present.
- Keep the methodology and exclusion scope in visible supporting text or a discoverable details region.
- Build these values from measured report data. Do not infer or invent code counts in browser JavaScript.

- Build the UI from the manifest's source-backed code index, not solely from rendered Markdown.
- Provide one search surface when the index has more than a handful of entries. Match model names, summaries, tags, symbols, relative paths, stages, categories, families, and learning phases.
- For a non-trivial index, expose all meaningful dimensions supplied by the Manifest: category (`categoryFilter`), pipeline stage (`codeStageFilter`), Symbol kind (`codeKindFilter`), model family (`codeFamilyFilter`), importance (`codeImportanceFilter`), learning phase (`codeLearningFilter`), and top-level path (`codePathFilter`). Do not remove a dimension merely to make the toolbar visually smaller.
- Add a clearly labeled core-only toggle (`coreOnly`) and enable it by default for newcomer orientation when core/important annotations exist. All Symbols remain reachable by disabling it.
- Add sorting (`codeSort`) with at least learning order, relevance, and path. Search relevance may be lightweight but exact Symbol/name matches must precede path-only matches.
- Use a compact filter toolbar with a primary row plus a discoverable secondary filter region when space is limited. Show option counts, disable zero-result options only when doing so cannot trap the user, and preserve selected state.
- Show result count, active filter summary (`activeFilters`), explicit empty state, and one-action reset (`clearCode`). Clearing filters must also reset search, core-only mode, sorting, and result pagination to documented defaults.
- Each result must expose repository-relative path, symbol, summary/responsibility when available, role/category, pipeline stage, model family, importance, learning phase, Symbol kind, language, and line number when known. Use progressive disclosure for secondary metadata.
- Source chips must support the navigation behavior in `references/html-code-navigation.md`: normal click fallback, copy feedback, and optional IDE/open actions.
- Use a brief toast or inline status for copy success/failure. Do not rely on changing the button label in a way that shifts layout.

### Model atlas as a learning map

- Group or filter models by architectural family rather than showing one flat grid.
- A model detail must answer: problem solved, architectural idea, prerequisites, key shared layers, example, test, source, and why it is worth learning.
- Offer a compact comparison mode for representative models so newcomers can contrast interaction mechanism, sequence support, multi-task support, complexity, and likely industrial stage.
- Keep the default atlas curated to representative models; provide an obvious path to the complete inventory.

### Learning route and change guide

- Render reading tracks as ordered, source-linked steps with estimated depth (`skim`, `read`, `deep-read`) and the question answered at each stop.
- Render modification intents as a task-oriented matrix: intent, start Symbol, related abstractions, regression test, and hidden contract/risk.
- Selecting a model, pipeline stage, reading step, or modification intent should be able to seed the Code Map filters without losing the user’s place.

## 7. Interaction Behavior

- Make controls feel immediate: hover, pressed, selected, focus, disabled, empty, and error states must be visually distinct.
- Keep motion between `140ms` and `220ms` for ordinary transitions. Animate opacity and transforms where possible; avoid height animations that create large layout shifts.
- Persist the chosen theme in `localStorage` and initialize from `prefers-color-scheme` when no preference exists.
- Clicking a TOC entry must immediately set its active style and `aria-current="location"` before smooth scrolling begins.
- Keep the active TOC item synchronized by recomputing against the full ordered section list and a fixed-header activation line. `IntersectionObserver` may trigger recomputation, but do not select only from entries delivered in one callback; provide a stable scroll-position fallback.
- Route click, scroll, `hashchange`, direct deep-link initialization, keyboard activation, and document-bottom handling through the same active-section function.
- Preserve deep links through section IDs. Opening a hash target inside a collapsed region must reveal that region.
- Do not implement interactions that hide essential content with no discoverable recovery path.
- Escape source-derived strings before inserting them with `innerHTML`, or construct nodes with safe DOM APIs.

## 8. Desktop Layout

Desktop is the default and required delivery target. Verify at minimum:

| Viewport | Expected layout |
|---|---|
| `1440 x 900` | Persistent navigation, full pipeline, multi-column model/code exploration |
| `1280 x 720` | Persistent navigation, compact but readable controls, no content hidden behind fixed regions |

- Use stable grid tracks, `minmax()`, and bounded widths so dynamic labels do not resize the interface.
- Keep dense tables, code paths, and flow diagrams readable without whole-page horizontal overflow.
- Tablet/mobile layout, touch-target tuning, drawers, and narrow-screen table conversion are out of the default scope. Add them only when the user explicitly requests responsive delivery.
- Never allow headings, path chips, buttons, arrows, floating controls, or tooltips to overlap or clip.

## 9. Accessibility and Motion

- Use semantic landmarks: `header`, `nav`, `main`, `section`, and `footer` where appropriate.
- Give icon-only controls accessible names. Expose expanded, selected, pressed, and live-update states with ARIA only when native semantics are insufficient.
- Provide a visible `:focus-visible` treatment with sufficient contrast.
- Maintain WCAG AA contrast for normal text and meaningful controls in both themes.
- Ensure the full report can be navigated with a keyboard. Focus order must follow visual order.
- Respect `prefers-reduced-motion: reduce` by disabling non-essential transitions and smooth scrolling.
- Use an `aria-live="polite"` region for search result counts, copy feedback, and dynamically changed pipeline details when useful.

## 10. Offline and Security Constraints

- Deliver one HTML file with inline CSS and JavaScript unless the user explicitly requests another format.
- Do not depend on CDNs, remote fonts, external icon libraries, remote JavaScript, or network-only assets.
- Keep the report useful when opened through `file://` in a mainstream browser.
- Do not claim that static HTML can execute local commands or reveal locations in an operating-system file manager. Keep primary copy platform-neutral and follow `references/html-code-navigation.md` for optional platform examples.
- Avoid `eval`, dynamic script injection, and unsanitized source-derived HTML.

## 11. Browser Verification

First declare the available QA level:

- **Browser QA**: a browser-control capability exists. Open the generated file directly and perform all checks below.
- **Static QA**: no browser-control capability exists. Run `scripts/validate_reports.py`, manually inspect the generated source, and state that visual/browser interaction QA was not executed.
- **Disclosed-only QA**: neither browser control nor Python is available. Perform available structural checks and list every skipped check.

For Browser QA, verify all of the following:

1. Capture or inspect the required desktop views at `1440 x 900` and `1280 x 720`.
2. Confirm the first viewport communicates repository identity, domain, and pipeline coverage.
3. Exercise theme switching, every TOC entry, drawer navigation, pipeline selection, keyboard pipeline navigation, search, filters, clear action, collapsible details, copy feedback, and back-to-top. For each TOC entry, verify both landing position and active state.
   Exercise every Code Map dimension, core-only mode, all sort choices, active-filter removal, zero-result recovery, and cross-links from model/learning/change sections.
4. Confirm source chips show correct paths and that their configured navigation behavior is honest about browser limits.
5. Inspect long headings, the longest paths, language distribution and role-scale statistics, wide tables, empty search results, absent pipeline stages, and dark mode.
6. Check the browser console for errors and the network panel for unintended remote requests.
7. Fix blank panels, stale transitions, overlap, clipping, horizontal page overflow, unreadable contrast, and controls hidden beneath fixed UI.

## 12. Acceptance Checklist

The HTML is complete only when:

- [ ] It behaves as a reading tool, not a decorated Markdown dump.
- [ ] Pipeline and data flows are visual, interactive where useful, and source-anchored.
- [ ] Models and code locations are searchable or easily filterable when the index is non-trivial.
- [ ] Non-trivial Code Maps expose category, stage, kind, family, importance, learning phase, path root, core-only, sorting, counts, active filters, and reset without hiding access to all Symbols.
- [ ] Model atlas, learning route, and change guide answer problem/architecture/prerequisite/example/test/extension questions with source anchors.
- [ ] Navigation remains available and stable at both required desktop widths.
- [ ] Every TOC entry has correct immediate and scroll-synchronized active state, including direct hashes and the final section.
- [ ] Language/code-volume statistics match Markdown, remain readable at desktop widths, and disclose method/exclusions.
- [ ] All primary interactions work with mouse and keyboard.
- [ ] Light/dark themes, focus states, empty states, and feedback states are polished.
- [ ] The page has no external runtime dependency, console error, unintended network request, overlap, clipping, or whole-page horizontal overflow.
- [ ] The completed QA level is recorded honestly; Browser QA includes `1440 x 900` and `1280 x 720`.
